# Secrets Inventory and Recovery

Every secret the cluster uses today: where it comes from, whether you
need to save it in a password manager, and how to recreate it if lost.
Companion to `docs/deploy.md` (build-time CI secrets) and
`infra/platform/README.md` / per-chart READMEs (chart bootstrap commands).

This file is the answer to "did I forget to save something?". The short
version: keep your **Hetzner Cloud** and **Cloudflare** account access
solid, and you can rebuild everything else from scratch in under an hour.

## Three categories of secrets

**Category A — externally issued.** API tokens minted in a cloud
provider's dashboard (Hetzner, Cloudflare). The dashboard is the source
of truth; the cluster holds a copy. **If lost: regenerate in the
dashboard, recreate the k8s secret.** No data loss.

**Category B — operator-managed.** Generated and rotated by an in-cluster
operator (CNPG for Postgres credentials, cert-manager for TLS). Lifecycle
is automatic. **You don't need to save these.** A cluster restore from
backup brings them back.

**Category C — one-off random.** Generated once with `openssl rand`
during a chart's bootstrap (Matomo's MariaDB root password, the
analytics ID salt). The k8s secret IS the source of truth. **Save these
in a password manager when you generate them.** If lost: reset by
generating new values, but you also lose access to the data they
guard (e.g., losing MariaDB's root password means rebuilding the DB).

## What to put in your password manager today

Three things, even if they're not lost:

1. **Hetzner Cloud account** — login + 2FA recovery codes.
2. **Cloudflare account** — login + 2FA recovery codes.
3. **Terraform state bucket access** — the access key referenced in
   `terraform/`'s remote state config (ADR-0010 §1). This is a
   Hetzner Object Storage credential, regenerate-able, but the key
   currently in your tofu config is the one your local has.

That's the minimum. Everything else flows from those.

When you bootstrap Matomo (Category C secrets land), add an entry per
generated password. The `infra/matomo/README.md` bootstrap snippet uses
shell variables — copy them into the password manager before you run
`helm install`.

## Bootstrapping a new cluster from zero

If the cluster is destroyed and you need to rebuild:

1. **Cluster provisioning.** `terraform apply` from `terraform/k8s/` —
   spins up Hetzner nodes, attaches floating IP, installs k3s.
2. **Platform secrets.** Recreate `cloudflare-api-token` and
   `hcloud-csi-token` in the `platform` namespace from your dashboard
   tokens. Commands in `infra/platform/README.md`.
3. **Platform chart.** `helm install platform infra/platform/ …` —
   brings up cert-manager, ingress-nginx, external-dns, CNPG operator,
   hcloud-csi.
4. **Application charts.** Recreate per-app env secrets
   (`wordsparrow-api-env`, `bliss-game-api-env`, `cnpg-backup-creds`),
   then `helm install` each app chart. The CD pipeline takes over from
   the next push to `main`.
5. **CNPG restore.** With `cnpg-backup-creds` in place, CNPG bootstraps
   the cluster from the latest barman-managed backup in Object Storage.
   App data returns intact.
6. **Matomo (if deployed).** New random passwords, fresh DB. No data
   restore unless you've added a MariaDB backup workflow (currently
   tracked as follow-up in `infra/matomo/README.md`).

Estimated time on a clean Hetzner account: 30–60 minutes if you have
all the tokens; longer if you have to mint them.

## When a secret rotates

Rotation is the same shape as recovery, just intentional:

```sh
# Example: rotate the Cloudflare token after a suspected leak.
NEW_TOKEN=$(read-from-cloudflare-dashboard)
kubectl -n platform delete secret cloudflare-api-token
kubectl -n platform create secret generic cloudflare-api-token \
  --from-literal=cloudflare_api_token="$NEW_TOKEN"
kubectl -n platform rollout restart deploy/external-dns  # picks up the new value
```

Rolling restarts are sufficient for envFromSecret-based secrets
(application picks up new values on pod start).

## What's deliberately out of scope

- Sealed-secrets / External Secrets Operator. Considered but not
  adopted: the secret count is small enough that one-time
  `kubectl create secret` per cluster is the simpler and auditable path
  (ADR-0009 §10). Revisit if the secret count grows past ~10 per
  cluster.
- Centralised vault (HashiCorp Vault, Bitwarden Secrets Manager,
  Doppler). Same reasoning — solo-dev scale.
- Backup-of-secrets-themselves. Category A and B are recoverable; only
  Category C needs offline storage and that's what your password
  manager is for. Backing up `kubectl get secret -o yaml` would be
  copying secret material outside the cluster's blast radius — strictly
  worse than the password-manager flow.
