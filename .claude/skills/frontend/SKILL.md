# Frontend playbook

Use this when your task lives under `frontend/`. The conventions below are binding — they're enforced by lint, tests, or CI gates. Don't relitigate them in a PR.

## Anchor documents

- `docs/adr/0002-frontend-stack.md` — the binding choice of Vite + React 19 + TanStack Router + Panda CSS + Vitest + MSW. §4 codifies the **uncontrolled-input** contract that all the grid-cell handlers depend on.
- `frontend/eslint.config.js` — eslint-plugin-boundaries enforces the hexagonal layering. If your import crosses a layer boundary, the lint job fails — fix the layer, don't disable the rule.
- `frontend/panda.config.ts` — Panda CSS tokens (colors, spacing, fontSizes, fonts).
- `docs/adr/0005-brand-identity.md` — color tokens, the `leaf.700` rule, etc.

## Stack at a glance

| Concern | Choice | Notes |
|---|---|---|

| Routing | TanStack Router | File-based routes under `src/ui/routes/` |

| Node version | ≥22 | Pinned in `package.json` engines |

## Hexagonal layers

```
src/
├── domain/             # pure types + invariants. No React, no fetch, no DOM.
├── application/        # ports (interfaces) + use cases. No React.
├── infrastructure/     # adapters (HTTP clients, localStorage, MSW)
└── ui/                 # React components + routes + Panda styles
```

eslint-plugin-boundaries enforces:
- `domain` imports nothing.
- `application` imports `domain` only.
- `infrastructure` imports `application` and `domain`.
- `ui` imports anything below.

If you need to cross a boundary, you're modeling something wrong — usually the missing piece is a port in `application` that an adapter in `infrastructure` implements and `ui` calls.

The composition root is `src/main.tsx`. It is the **only** file allowed to wire concrete adapters into the router context. Everything else receives them via the TanStack Router context (`createAppRouter(context)`).

## ADR-0002 §4: inputs are uncontrolled

This is the most important convention to internalize. Letter cells in the grid are uncontrolled `<input>` elements:

- No `value={...}` + `onChange={...}` round-tripping through React state.
- `defaultValue={cell.entry}` for the initial value; subsequent writes go directly via `el.value = ...` inside the handlers.
- A keystroke writes synchronously, advances focus, broadcasts (via the optional `onCellChange` prop wired in PR #124) — without a re-render in the path.

Why: the grid has dozens to hundreds of inputs, and the controlled pattern triggers a top-level re-render on every keystroke. Uncontrolled keeps the keystroke→glyph latency under one frame even on weak phones.

Implications:
- Don't lift cell values into React state. If you need the current value of a cell, read it from the DOM via the `refs` map in `useGridNavigation`.
- Don't add `value={…}` to letter cells. eslint-plugin-react has no rule against this; the only safety net is the test suite + PR review.
- For pasted multi-char content / non-letter input, the truncation rule is enforced inside `handleInput` (sets `target.value` to a normalized single uppercase letter or `''`) — see `useGridNavigation.ts`.

## TanStack Router context

`src/ui/router.ts` exports `createAppRouter(context)`. The context object carries adapters (currently `puzzleRepository`; in Wave H it'll also carry `gameClient`, `lobbyClient`, `session`).

Routes access context via the standard TanStack hooks:

```tsx
const { puzzleRepository } = Route.useRouteContext();
```

Don't import the concrete adapter directly from a route; always go through context. This is what lets MSW (in preview mode) and Vitest tests substitute fakes.

## Generated API types

`src/infrastructure/api/<context>/types.ts` is generated by `pnpm api:generate` (which runs `openapi-typescript` against the OpenAPI spec). Never edit it by hand — the CI `regen-and-diff` job re-runs the generator and fails if the file drifts.

Workflow when an OpenAPI spec changes:
1. Schema PR lands the OpenAPI change.
2. In the consumer PR, run `cd frontend && pnpm api:generate`.
3. Commit the regenerated `types.ts`. ADR-0001 §4 excludes generated code from the line cap, so the diff doesn't count against you.

## MSW (preview mode only)

`src/infrastructure/mocks/` defines handlers for preview deploys (`VITE_USE_MOCK_API=true`). Production builds (`VITE_USE_MOCK_API=false`) tree-shake the entire MSW module — verified periodically with:

```
pnpm build && grep -r setupWorker dist/   # → empty
```

Don't import MSW from any non-preview code path. Don't add MSW handlers to the prod bundle.

## Test patterns

- Place tests under `frontend/tests/`. Co-located `__tests__/` under `src/` is not used in this project.
- Vitest globals are off; import `describe / it / expect / vi / beforeEach / afterEach` explicitly.
- Use Testing Library queries for accessible-by-design assertions:
  ```ts
  screen.getByRole('grid', { name: ... });
  screen.getByLabelText('définition horizontale');
  ```
- For low-level grid tests, query by `data-row` / `data-col` / `data-cell-kind`:
  ```ts
  container.querySelector(`[data-cell-kind="letter"][data-row="${row}"][data-col="${col}"]`);
  ```
- Wrap state-mutating events in `act(() => ...)` so React 18+ flushes the update before your assertions run.
- jsdom doesn't auto-focus on `fireEvent.click`. The standard helper is `const click = (el) => { el.focus(); fireEvent.click(el); };`.
- Some browser APIs are missing in jsdom; `vitest.setup.ts` polyfills `window.scrollTo`, `window.scrollBy`, and `ResizeObserver`. Add new polyfills there, not per-test.

## Validation commands

Run all four before pushing — CI runs all four:

```
cd frontend
pnpm typecheck
pnpm lint
pnpm test --run tests/<your-file>.test.tsx   # focused
pnpm test                                     # full suite
pnpm build
```

## Known pre-existing failure

`tests/http-puzzle-repository.test.ts` expects `puzzle.width === 5` but the canonical fixture is 10×10 since #106. This fails on every PR. Note in your PR body as known-pre-existing-on-main; do not fix from an unrelated PR.

## Common failure modes

| Symptom | Cause | Fix |
|---|---|---|
| `regen-and-diff` CI fails | OpenAPI changed but `types.ts` not regenerated | `cd frontend && pnpm api:generate`; commit the diff. |
| Lint fails with "import boundary violated" | Crossed a layer (e.g. `ui` imported from `infrastructure` directly) | Add a port in `application`; have `infrastructure` implement it; pass via router context. |
| Vitest hangs / leaks | Missing `act()` around a state update; lingering listener | Wrap in `act`, ensure `useEffect` cleanups remove all listeners. |
| `ResizeObserver is not defined` in jsdom | A library uses it; jsdom doesn't ship it | Add a no-op polyfill to `vitest.setup.ts`. |
| Bundle bigger than expected | New dep didn't tree-shake | Verify with `pnpm build` locally; check the new dep has `sideEffects: false` or imports a single export, not a barrel. |
| CSS token doesn't resolve | Used a hex literal instead of a Panda token | Use `token(colors.leaf.700)` — Panda extracts at build time. Hex literals bypass the design system. |

## Don'ts

- **Don't** lift letter-cell values into React state. Uncontrolled per ADR-0002 §4.
- **Don't** import MSW outside preview-mode code paths.
- **Don't** edit generated `types.ts` by hand.
- **Don't** add `eslint-disable` for boundary violations — fix the layering.
- **Don't** pin a dep that ships its own React. We're on React 19; deps should peer-depend on it.
- **Don't** add a barrel `index.ts` to a new infra subfolder — none of the existing ones use them, and they hurt tree-shaking.
- **Don't** import from `frontend/src/...` with deep relative paths from tests; use the `@/` alias resolved by `vite.config.ts` and `tsconfig.json`.
- **Don't** mutate `process.env` in tests. Use `vi.stubEnv(...)` if you must.
- **Don't** call setState inside a render body. Move it into `useEffect` or a callback.
